# CMS-Core Feature Specification

## Project Overview

**Name:** CMS-Core
**Type:** Zero-dependency Node.js CMS with Drupal-inspired architecture
**Version Target:** 1.0.0
**Existing Codebase:** /Users/Alchemy/Projects/experiments/cms-core (v0.0.80)

This specification defines features to bring CMS-Core to full Drupal parity, based on comprehensive analysis of Drupal's 75 core modules.

## Technology Stack

- **Runtime:** Node.js 20+ (ES modules)
- **Dependencies:** ZERO npm packages (all built from node: built-ins)
- **Storage:** Flat-file JSON
- **Server:** node:http
- **Architecture:** Service pattern (init/register exports)

## Core Constraint

**CRITICAL:** This project has NO npm dependencies by design. All functionality must be built using only Node.js built-in modules:
- node:http, node:https
- node:fs, node:path
- node:crypto
- node:url
- node:stream
- node:events

## Feature Count

**feature_count: 165**

This is a MEDIUM complexity project targeting Drupal parity.

## TIER 1: INFRASTRUCTURE (Features 0-4)

These must pass before any functional features.

### Feature 0: Server starts without errors
- Start server with `node index.js`
- Verify no errors in console
- Server responds on http://localhost:3001

### Feature 1: CLI framework operational
- Run `node index.js help`
- Verify 300+ commands listed
- Commands execute without errors

### Feature 2: Content storage persistence
- Create content via CLI: `node index.js content:create article '{"title":"TEST_12345"}'`
- Stop server completely
- Restart server
- Verify content still exists via `node index.js content:list --type=article`

### Feature 3: Configuration loading
- Read config/site.json
- Read config/modules.json
- All registered modules initialize

### Feature 4: Service architecture intact
- All 91 core services load
- Services register CLI commands
- Hook system functional

## TIER 2: PENDING REVISIONS (Features 5-24)

Based on Drupal's content_moderation module. Allows draft edits on published content.

### Core Concept
- `published` flag = visible to public
- `isDefaultRevision` flag = canonical version
- These are INDEPENDENT, enabling "pending draft on published content"

### Features

5. Add isDefaultRevision field to revision metadata
6. Content.get() returns default revision by default
7. Content.get() accepts includeAllRevisions option
8. Content.list() returns default revisions only by default
9. Content.list() accepts includeAllRevisions option
10. New revision inherits isDefaultRevision from workflow state
11. Draft state creates non-default revision on published content
12. Publish transition makes revision the new default
13. CLI: content:revisions shows default flag
14. CLI: content:set-default sets default revision
15. API: GET /jsonapi returns default revisions only
16. API: GET /jsonapi?include=allRevisions returns all
17. Pending revision indicator in content list
18. Pending revision count per content item
19. Discard pending revision action
20. Compare pending to published revision
21. Workflow state reflects pending status

23. Bulk publish pending revisions
24. Audit log tracks default revision changes
