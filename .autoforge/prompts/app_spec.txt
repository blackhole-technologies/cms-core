# CMS-Core Stage 1: Foundation Patterns
# The core patterns everything else depends on.
# Zero external dependencies. Node.js built-ins only.

## Architecture Context
CMS-Core is an existing Node.js CMS with ~90 core files and ~20 modules.
This stage adds 5 foundational pattern systems under core/lib/ that will
underpin ALL future extensibility. Read CLAUDE.md for full architecture guide.

## CRITICAL: Read Templates First
Before implementing any feature, read these template files:
- .autoforge/templates/plugin-manager.js — PluginManager pattern
- .autoforge/templates/plugin-base.js — PluginBase pattern
- .autoforge/templates/service-provider.js — ServiceProvider pattern
- .autoforge/templates/hook-implementation.js — Hook pattern
- .autoforge/templates/access-result.js — AccessResult pattern
- .autoforge/templates/config-entity.js — ConfigEntity pattern

## Technology Stack
- Runtime: Node.js (zero npm dependencies)
- Storage: JSON files + SQLite (via core/database.js)
- Server: Built-in HTTP (core/server.js)
- Port: 3001

## Phase 1.1: Unified Plugin System (Features 1-6)

1. Create PluginManager class in core/lib/Plugin/PluginManager.js with constructor(type, options) that accepts type string, subdir, alterHook, baseClass, and defaults options. Must store type, subdir, alterHook, baseClass, defaults as instance properties. Must have private _modulePaths array and null definition cache.
   ACCEPTANCE: File exists at core/lib/Plugin/PluginManager.js. Class is exported. Constructor sets all properties correctly.

2. Implement PluginManager.getDefinitions() as async method that scans each module path's plugins/{subdir}/ directory for .js files, dynamically imports each, reads the exported 'definition' object, merges with defaults, stores _module and _path metadata, and caches results in a Map keyed by definition.id. Must handle missing directories gracefully (ENOENT).
   ACCEPTANCE: Create modules/test_plugin/plugins/test_type/Alpha.js with definition {id:'alpha', label:'Alpha'}. Call getDefinitions(). Map contains 'alpha' key with merged definition.

3. Implement PluginManager.createInstance(id, configuration) that calls getDefinition(id), then uses the plugin's exported default/create factory function to instantiate with (configuration, id, definition, services) arguments. If no factory exists, return a PluginInstance wrapper. Throw descriptive error if plugin ID not found (list available IDs).
   ACCEPTANCE: createInstance('alpha') returns an object. createInstance('nonexistent') throws with available plugin IDs listed.

4. Implement PluginManager.clearCachedDefinitions(), hasDefinition(id), and setInfrastructure(services, hooks, modulePaths) methods. setInfrastructure stores references for alter hooks and module scanning.
   ACCEPTANCE: After clearCachedDefinitions(), next getDefinitions() re-scans filesystem. hasDefinition returns boolean.

5. Create PluginBase class in core/lib/Plugin/PluginBase.js with getPluginId(), getPluginDefinition(), getConfiguration() methods. Constructor takes (configuration, pluginId, pluginDefinition). Export class.
   ACCEPTANCE: File exists. new PluginBase({foo:1}, 'test', {id:'test'}).getPluginId() returns 'test'.

6. Create core/lib/Plugin/index.js barrel export that re-exports PluginManager and PluginBase. Verify the plugin system works end-to-end: create a test module with a plugin, discover it, instantiate it, verify alter hook fires via trigger on getDefinitions.
   ACCEPTANCE: import { PluginManager, PluginBase } from 'core/lib/Plugin/index.js' works. End-to-end test passes.

## Phase 1.2: Service Container (Features 7-11)

7. Create Container class in core/lib/DependencyInjection/Container.js with register(name, factory, options) method. Options support: deps (string array of dependency service IDs), tags (string array), singleton (boolean, default true), alias (string). Store definitions in a Map. Index by tags in a separate Map of tag→Set<serviceName>.
   ACCEPTANCE: container.register('foo', () => new Foo(), {tags:['my_tag']}). container._definitions has 'foo'.

8. Implement Container.get(name) that resolves aliases, resolves dependency services recursively, calls factory with resolved deps, caches singleton instances. Support optional dependencies prefixed with '?' that return null if missing. Throw descriptive error for unknown services (list available).
   ACCEPTANCE: Register A depending on B. get('A') resolves B first. get('?missing') returns null. get('unknown') throws with list.

9. Implement Container.getTagged(tag) returning array of {name, service} for all services with that tag. Implement has(name), list(), reset() (clears cached instances), and getLazy(name) that returns a Proxy deferring instantiation until first property access.
   ACCEPTANCE: Register 3 services tagged 'plugin_manager'. getTagged('plugin_manager') returns 3 entries. getLazy works.

10. Implement Container.registerProvider(moduleName, registrar) that calls registrar(container) — modules use this to register their services. Implement decorate(name, decorator) that wraps an existing service factory.
    ACCEPTANCE: registerProvider('node', (c) => c.register('node.storage', ...)). decorate wraps existing service.

11. Create core/lib/DependencyInjection/index.js barrel export. Write a ServiceProvider example in .autoforge/templates/service-provider.js showing module service registration pattern. Create Reference class in core/lib/DependencyInjection/Reference.js for declarative dependency references.
    ACCEPTANCE: All files exist. Imports work. ServiceProvider template has JSDoc and example.

## Phase 1.3: Unified Hook System (Features 12-16)

12. Create HookManager class in core/lib/Hook/HookManager.js with on(hookName, handler, options) method. Options: priority (number, default 10, lower runs first), module (string), once (boolean). Store handlers in a Map of hookName→sorted array. Return this for chaining.
    ACCEPTANCE: hooks.on('test', fn, {priority:5, module:'mymod'}). handlers stored and sorted by priority.

13. Implement HookManager.invoke(hookName, context) that runs all handlers sequentially (async), passing context. Remove once-handlers after firing. Log errors per handler but continue execution. Implement invokeAll(hookName, context) that collects return values from all handlers.
    ACCEPTANCE: Register 3 handlers for 'test'. invoke('test', {}) runs all 3 in priority order. Error in #2 doesn't stop #3.

14. Implement HookManager.alter(hookName, data, context) for data modification hooks. Runs handlers for '{hookName}_alter', each handler receives (data, context) and may return modified data. Implement onAlter(hookName, handler, options) shorthand.
    ACCEPTANCE: hooks.onAlter('form', (form) => ({...form, extra: true})). alter('form', {a:1}) returns {a:1, extra:true}.

15. Implement HookManager.remove(hookName, moduleName) and reorder(hookName, moduleName, newPriority). Add backward-compatible aliases: register() maps to on(), trigger() maps to invoke(), hasHandlers() checks handler existence.
    ACCEPTANCE: Existing code using hooks.register() and hooks.trigger() still works. remove/reorder modify handler list.

16. Create core/lib/Hook/index.js barrel export. Verify end-to-end: register multiple hooks from different "modules", invoke them, verify order, verify alter chains, verify once-handlers auto-remove.
    ACCEPTANCE: Integration test passes. Barrel export works.

## Phase 1.4: Config Entity System (Features 17-21)

17. Create ConfigEntity class in core/lib/Config/ConfigEntity.js with constructor(entityTypeId, values). Properties: uuid (auto-generated via crypto.randomUUID()), id, label, status (boolean), dependencies object, langcode. Methods: get(key), set(key, value), isDirty() using structuredClone comparison, toJSON(), getConfigName() returning '{entityTypeId}.{id}'.
    ACCEPTANCE: new ConfigEntity('node_type', {id:'article', label:'Article'}). isDirty() false initially, true after set().

18. Create ConfigEntityStorage class in core/lib/Config/ConfigEntityStorage.js with constructor(entityTypeId, configDir). Methods: load(id), save(entity), delete(id), loadMultiple(ids), loadAll(). Storage as JSON files in {configDir}/active/{entityTypeId}.{id}.json. In-memory cache.
    ACCEPTANCE: save(entity) writes JSON file. load(id) reads it back. loadAll() returns all for type. delete() removes file.

19. Implement ConfigEntityStorage.exportToStaging() that copies all config to {configDir}/staging/. This enables config export/import for deployment. Create directories automatically with mkdirSync({recursive:true}).
    ACCEPTANCE: exportToStaging() copies files to staging directory. Returns count of exported entities.

20. Create ConfigSchema class in core/lib/Config/ConfigSchema.js with validate(entity, schema) method that checks required fields, types, and constraints. Schema is a plain object defining expected structure. Return {valid, errors} object.
    ACCEPTANCE: validate({id:'test'}, {id:{type:'string',required:true}, label:{type:'string',required:true}}) returns errors for missing label.

21. Create core/lib/Config/index.js barrel export re-exporting ConfigEntity, ConfigEntityStorage, ConfigSchema. Create config/active/ and config/staging/ directories if they don't exist.
    ACCEPTANCE: All imports work. Directories exist. Full round-trip: create entity → save → load → modify → save → verify change persisted.

## Phase 1.5: Access Result System (Features 22-25)

22. Create AccessResult class in core/lib/Access/AccessResult.js with static factory methods: allowed(), forbidden(reason), neutral(reason). Instance methods: isAllowed(), isForbidden(), isNeutral(), getReason(). Use Symbol for private reason storage.
    ACCEPTANCE: AccessResult.allowed().isAllowed() === true. AccessResult.forbidden('no perm').getReason() === 'no perm'.

23. Implement AccessResult.orIf(other) and andIf(other) for combining results. Forbidden wins over Allowed; Allowed wins over Neutral. Implement cache metadata methods: addCacheContexts(array), addCacheTags(array), cachePerPermissions(), cachePerUser().
    ACCEPTANCE: allowed().orIf(forbidden('x')).isForbidden(). allowed().andIf(neutral()).isNeutral(). Cache methods chain.

24. Create combineAccessResults(resultsArray) exported function that reduces array of AccessResult using orIf logic. Short-circuit on first forbidden. Create AccessPolicy class in core/lib/Access/AccessPolicy.js with check(account, operation, context) method stub.
    ACCEPTANCE: combineAccessResults([allowed(), neutral(), forbidden('x')]).isForbidden().

25. Create core/lib/Access/index.js barrel export. Write integration test: simulate access check for entity view operation combining permission check + ownership check + admin override. Verify correct result.
    ACCEPTANCE: All imports work. Multi-policy access check returns correct combined result.
