# CMS-Core: ConfigEntityStorage
# Completes the config entity pattern — ConfigEntity exists but can't persist itself.
# Adds ConfigEntityStorage that reads/writes config/active/{entityTypeId}.{id}.json
# and integrates with EntityTypeManager for type registration.

## Architecture Context
CMS-Core has core/lib/Config/ConfigEntity.js implemented but it has no storage counterpart.
The config-entity.js template shows the full pattern including ConfigEntityStorage.
EntityTypeManager (being built separately) will use this for config entity type registration.
The template already has a working ConfigEntityStorage — this spec refines it to match
the core/lib/ code style and integrates it properly.

## Read These Files First
- core/lib/Config/ConfigEntity.js — existing ConfigEntity implementation
- .autoforge/templates/config-entity.js — full pattern with ConfigEntityStorage
- core/lib/Plugin/PluginManager.js — code style reference
- core/lib/Entity/EntityTypeManager.js — integration point (may be built concurrently)

## Code Style Rules (MUST follow — match existing core/lib/ files exactly)
- @file JSDoc header with Drupal equivalent reference
- const SYMBOL = Symbol('name') for ALL private/internal state at module top level
- WHY comments on every non-trivial decision
- Full JSDoc on every public method with @param and @return
- Descriptive error messages that list available values
- export class (NOT default exports)
- return this for chaining on mutating methods
- Zero npm dependencies — Node.js built-ins only, async I/O (not sync)
- ES module syntax (import/export)

## Technology Stack
- Runtime: Node.js (zero npm dependencies)
- Storage: JSON files under config/active/ and config/staging/
- Port: 3001

feature_count: 8

## Phase 1: ConfigEntityStorage Core (Features 1-4)

1. Create ConfigEntityStorage class in core/lib/Config/ConfigEntityStorage.js. Constructor takes (entityTypeId, configDir = 'config'). Store entityTypeId, compute activeDir as {configDir}/active and stagingDir as {configDir}/staging. Use async mkdir (from node:fs/promises) to ensure directories exist on first operation, NOT in constructor. Use Symbol-based private state. Add @file JSDoc header referencing Drupal ConfigEntityStorage.php.
   ACCEPTANCE: new ConfigEntityStorage('node_type', 'config') creates instance. No directories created until first load/save. File has @file JSDoc header.

2. Implement ConfigEntityStorage.save(entity) as async method. Validates entity has getConfigName() method (throws descriptive error if not). Writes JSON to config/active/{entityTypeId}.{id}.json using async writeFile. Creates active directory if missing. Caches entity in internal Map. Implement ConfigEntityStorage.load(id) as async method that reads from config/active/{entityTypeId}.{id}.json, creates new ConfigEntity from parsed JSON, caches and returns it. Returns null if file doesn't exist. Use async readFile.
   ACCEPTANCE: save() writes JSON file to correct path. load() reads it back. load('nonexistent') returns null. Cached on second load.

3. Implement ConfigEntityStorage.delete(id) as async method that removes config/active/{entityTypeId}.{id}.json and clears cache entry. Handles missing files gracefully (no throw). Implement loadMultiple(ids) that loads array of ids, returns Map of id→entity (skips nulls). Implement loadAll() that reads all files matching {entityTypeId}.*.json in active directory, returns Map of id→entity.
   ACCEPTANCE: delete() removes file and cache entry. delete('nonexistent') doesn't throw. loadMultiple(['a','b']) returns Map. loadAll() finds all matching files.

4. Implement ConfigEntityStorage.exportToStaging() as async method that copies all active config to config/staging/ directory. Creates staging directory if missing. Returns count of exported files. Implement importFromStaging() as async method that copies all staging config to active directory, clearing cache. Returns count of imported files. This is the deployment pipeline: export on source environment, import on target.
   ACCEPTANCE: exportToStaging() copies files from active to staging, returns count. importFromStaging() copies from staging to active, returns count.

## Phase 2: Query Support and Integration (Features 5-7)

5. Implement ConfigEntityStorage.executeQuery(query) to support EntityQuery. Reads query._conditions, query._sorts, query._range, query._count from EntityQuery accessors. Loads all entities via loadAll(), filters by conditions (supports =, <>, IN, STARTS_WITH, CONTAINS operators on entity values), sorts by field/direction, applies range pagination, returns array of entity IDs (or count if query._count is true). This makes ConfigEntityStorage compatible with EntityTypeManager.getQuery().
   ACCEPTANCE: With entities saved, executeQuery with condition('status', true) returns matching IDs. With .count() returns number. Sorting and range work correctly.

6. Implement ConfigEntityStorage.create(values) that returns a new ConfigEntity instance with the entityTypeId and provided values. Does NOT save — just creates in memory. If values has no uuid, one is generated. If values has no id, entity.id remains null until set. This matches the Drupal pattern: $storage->create($values) then $entity->save().
   ACCEPTANCE: create({id: 'article', label: 'Article'}) returns ConfigEntity with those values. create({}) returns ConfigEntity with generated uuid and null id. Entity is not persisted until save() is called.

7. Create core/lib/Config/index.js barrel export that re-exports ConfigEntity from './ConfigEntity.js' and ConfigEntityStorage from './ConfigEntityStorage.js'. Follow same pattern as core/lib/Plugin/index.js.
   ACCEPTANCE: import { ConfigEntity, ConfigEntityStorage } from './core/lib/Config/index.js' works. Both classes available.

## Phase 3: Verification (Feature 8)

8. End-to-end integration test: Create ConfigEntityStorage for 'node_type' entity type. Use create() to make entities for 'article' and 'page' with labels and status. Save both. Load by ID. LoadAll. Query with condition('status', true). Export to staging. Import from staging. Delete one. Verify all operations work correctly and files are created/removed at the right paths under config/active/ and config/staging/.
   ACCEPTANCE: Full round-trip works: create → save → load → query → export → import → delete. Files exist at config/active/node_type.article.json etc. Query returns correct filtered results.
