# CMS-Core: EntityQuery + EntityTypeManager
# The missing foundation of the entity system.
# These two classes complete the entity pattern triad alongside the existing templates.

## Architecture Context
CMS-Core is an existing Node.js CMS with ~90 core files and ~20 modules.
4 of 8 meta-patterns are implemented in core/lib/ (PluginManager, Container, HookManager, ConfigEntity).
The Entity subsystem has NO core/lib/Entity/ directory yet. These features create it.

Read these existing files for code style and patterns:
- core/lib/Plugin/PluginManager.js — Symbol-based private state, JSDoc, WHY comments, setInfrastructure pattern
- core/lib/DependencyInjection/Container.js — Dependency resolution, error messages listing available values
- core/lib/Hook/HookManager.js — Async invocation, priority ordering
- core/lib/Config/ConfigEntity.js — Entity-like value objects

Read these templates for the EntityQuery/EntityTypeManager contract:
- .autoforge/templates/entity-storage.js — Shows how storage.executeQuery(query) is called, and query._conditions/_sorts/_range/_count accessors
- .autoforge/templates/content-entity.js — Shows entityType.keys.id, entityType.keys.uuid etc.

## Code Style Rules (MUST follow — match existing core/lib/ files exactly)
- @file JSDoc header with Drupal equivalent reference and @see template path
- const SYMBOL = Symbol('name') for ALL private/internal state at module top level
- WHY comments on every non-trivial decision
- Full JSDoc on every public method with @param and @return
- Descriptive error messages that list available values (see Container.get() and PluginManager.createInstance())
- export class (NOT default exports)
- return this for chaining on mutating methods
- Zero npm dependencies — Node.js built-ins only
- ES module syntax (import/export)
- Async methods where I/O is involved

## Technology Stack
- Runtime: Node.js (zero npm dependencies)
- Storage: JSON files (core/content.js is current storage)
- Port: 3001

## Phase 1: EntityQuery — Fluent Query Builder (Features 1-5)

1. Create ConditionGroup class in core/lib/Entity/EntityQuery.js. Constructor takes conjunction parameter ('AND' or 'OR', default 'AND'). Implement .condition(field, value, operator = '=') that validates operator against static VALID_OPERATORS set (=, <>, !=, >, >=, <, <=, IN, NOT IN, BETWEEN, IS NULL, IS NOT NULL, STARTS_WITH, CONTAINS, ENDS_WITH) and throws immediately with full operator list if invalid. Normalize != to <> internally. Store conditions as array of {field, value, operator} objects. Implement .exists(field) as shorthand for .condition(field, null, 'IS NOT NULL') and .notExists(field) as shorthand for .condition(field, null, 'IS NULL'). Implement .addGroup(conditionGroup) to nest groups. Implement .getConditions(), .getGroups(), .getConjunction() accessors. All mutating methods return this for chaining. Use Symbol-based private state for conditions array, groups array, and conjunction. Add WHY comments. Add full JSDoc.
   ACCEPTANCE: new ConditionGroup('AND').condition('status', true).condition('type', 'article').getConditions() returns [{field:'status',value:true,operator:'='},{field:'type',value:'article',operator:'='}]. new ConditionGroup().condition('x', 'y', 'INVALID') throws error listing all valid operators. new ConditionGroup().condition('x', 'y', '!=').getConditions()[0].operator === '<>'.

2. Create EntityQuery class in the same file core/lib/Entity/EntityQuery.js. Constructor takes (entityTypeId, storage, options = {}). entityTypeId is required string. storage is required object — validate it has an executeQuery method or throw. options.accessHandler is optional. Store all state via Symbols. Create a root ConditionGroup('AND') internally. Implement fluent API: .condition(field, value, operator) delegates to root group. .exists(field) and .notExists(field) delegate to root group. .sort(field, direction = 'ASC') validates direction is ASC or DESC, stores in sorts array. .range(start, length) stores pagination. .count() sets count mode flag. .accessCheck(enabled) takes boolean, stores flag. All return this for chaining.
   ACCEPTANCE: new EntityQuery('node', mockStorage).condition('type','article').sort('created','DESC').range(0,10) chains without error. Constructor throws if storage lacks executeQuery method. sort with invalid direction throws.

3. Implement EntityQuery.andConditionGroup() and .orConditionGroup() methods that create a new ConditionGroup with the respective conjunction, add it to the root group via addGroup(), and return the NEW group (not the query — this breaks the chain intentionally so conditions are added to the group). This enables: const or = query.orConditionGroup(); or.condition('a', 1).condition('b', 2); then continue with query.condition('c', 3).
   ACCEPTANCE: const q = new EntityQuery('node', mockStorage).accessCheck(false); const or = q.orConditionGroup(); or.condition('status','published').condition('status','featured'); q.condition('type','article'); q._rootGroup.getGroups().length === 1; q._rootGroup.getGroups()[0].getConjunction() === 'OR'.

4. Implement EntityQuery.execute() as async method. If accessCheck() was never called (flag is undefined, not false), log a console.warn: 'Calling EntityQuery.execute() without explicitly calling .accessCheck() is deprecated.' Then delegate to storage.executeQuery(this) and return the result. Implement read-only accessors as getters: _conditions (flat array from root group getConditions()), _sorts, _range ({start,length} or null), _count (boolean), _rootGroup (the full ConditionGroup tree), _entityTypeId (string). These are what storage handlers read to build actual queries.
   ACCEPTANCE: With mock storage that returns ['id-1','id-2'], new EntityQuery('node', mockStorage).accessCheck(false).condition('type','article').execute() resolves to ['id-1','id-2']. With mock that checks query._count and returns 42 when true, .count().execute() resolves to 42. Calling execute() without accessCheck() logs a warning.

5. Add @file JSDoc header to core/lib/Entity/EntityQuery.js with Drupal equivalent reference (EntityQuery.php, QueryInterface.php). Add @see .autoforge/templates/entity-storage.js. Export both ConditionGroup and EntityQuery classes (named exports, no defaults). Ensure the file follows the exact same code style as core/lib/Plugin/PluginManager.js — Symbol-based private state at module top, WHY comments on non-trivial decisions, descriptive errors.
   ACCEPTANCE: File has @file JSDoc header mentioning Drupal EntityQuery.php. Both classes are named exports. File uses Symbol() for private state. import { EntityQuery, ConditionGroup } from './core/lib/Entity/EntityQuery.js' works.

## Phase 2: EntityTypeManager — Type Registry + Handler Resolution (Features 6-9)

6. Create EntityType class in core/lib/Entity/EntityTypeManager.js as an exported value object. Constructor takes (id, definition). Store id as string. Extract from definition: label (string, defaults to id), keys object with defaults {id: 'id', uuid: 'uuid', bundle: null, label: null, revision: null}, revisionable (boolean, default false), translatable (boolean, default false), handlers (object, default {}), baseFieldDefinitions (object, default {}). Implement: .getKey(keyName) returns value or null, .hasKey(keyName) returns boolean, .hasHandler(handlerType) returns boolean, .toJSON() returns plain object. Use Symbol-based private state. WHY comments. Full JSDoc.
   ACCEPTANCE: const et = new EntityType('node', {label:'Content', keys:{id:'nid',uuid:'uuid',bundle:'type',label:'title'}, handlers:{storage:mockStorage}}). et.id === 'node'. et.label === 'Content'. et.keys.id === 'nid'. et.keys.bundle === 'type'. et.keys.revision === null. et.hasHandler('storage') === true. et.hasHandler('view_builder') === false. et.toJSON() returns plain object.

7. Create EntityTypeManager class in the same file core/lib/Entity/EntityTypeManager.js. Implement setInfrastructure(services, hooks) matching PluginManager pattern — stores Container and HookManager references. Implement register(entityTypeId, definition) that wraps definition in EntityType and stores in internal Map. Returns this for chaining. Implement getDefinition(entityTypeId) returning EntityType or null. Implement getDefinitions() returning Map of all types. Implement hasDefinition(entityTypeId) returning boolean. Use Symbol-based private state. Internal handler cache Map and storage cache Map.
   ACCEPTANCE: const mgr = new EntityTypeManager(); mgr.register('node', {label:'Content',keys:{id:'nid'}}); mgr.hasDefinition('node') === true. mgr.getDefinition('node').label === 'Content'. mgr.getDefinition('unknown') === null.

8. Implement EntityTypeManager.getStorage(entityTypeId) that gets the EntityType definition (throws descriptive error with available type IDs if not found), checks handlers.storage, resolves it: if object use directly, if function call as factory, if string resolve as service ID from Container (throw "Cannot resolve handler '${handlerId}' — Container not set. Call setInfrastructure() first." if Container is null). Cache resolved storage. Implement getHandler(entityTypeId, handlerType) with same resolution logic but returns null instead of throwing for missing handlers. Implement getQuery(entityTypeId) that calls getStorage(), then returns new EntityQuery(entityTypeId, storage, {accessHandler}) where accessHandler is resolved via getHandler(entityTypeId, 'access').
   ACCEPTANCE: With object handler: mgr.register('node', {handlers:{storage:mockStorage}}); mgr.getStorage('node') === mockStorage. With string handler and no infrastructure: throws about Container not set. mgr.getQuery('node') returns EntityQuery instance. mgr.getStorage('nonexistent') throws with available type IDs listed.

9. Implement EntityTypeManager.discoverEntityTypes() as async method that fires hook 'entity_type_info' via hooks.invokeAll() to collect definitions from modules, then fires 'entity_type_info_alter' via hooks.alter() to let modules modify them, then registers all discovered types. Make getDefinition() and getDefinitions() trigger lazy discovery on first call if hooks are wired (setInfrastructure was called with hooks) and cache hasn't been built yet. Implement clearCachedDefinitions() that resets the definitions Map, handler cache, and storage cache. Add @file JSDoc header with Drupal equivalent (EntityTypeManager.php, EntityTypeManagerInterface.php). Export both EntityType and EntityTypeManager.
   ACCEPTANCE: With mock hooks that return [{id:'node',label:'Content'}] from entity_type_info, discoverEntityTypes() populates definitions. clearCachedDefinitions() empties everything. File has @file JSDoc. Both classes are named exports.
