## Architecture Notes

**Why Integration Needed:**

The token fallback system (Features #1-3) implemented the core logic but needed integration into the main token replacement pipeline. Feature #4 completed this integration by:

1. **Pattern Detection**: Uses regex to find curly-brace patterns
2. **Dual Processing**: Handles both fallback tokens and standard tokens
3. **Sequential Processing**: Processes fallback tokens first, then standard tokens
4. **Shared Context**: Both systems use same context object for resolution

**Why Documentation Needed:**

Feature #5 ensures developers and content administrators understand:
- **Syntax**: How to write fallback tokens correctly
- **Behavior**: What happens when values are empty/missing
- **Performance**: Performance characteristics and limits
- **Best Practices**: How to use fallbacks effectively
- **Migration**: How to upgrade from standard tokens
