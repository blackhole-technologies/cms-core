# CMS-Core: Boot Bridge
# Wires the new core/lib/ patterns into boot.js and deprecates the legacy systems.
# This is the migration path from core/hooks.js → HookManager, core/services.js → Container.

## Architecture Context
CMS-Core has two parallel systems:
- Legacy: core/hooks.js, core/services.js, core/dependencies.js — used by boot.js and all modules
- New: core/lib/Hook/HookManager.js, core/lib/DependencyInjection/Container.js, core/lib/Plugin/PluginManager.js, core/lib/Entity/ (being built)

boot.js currently uses ONLY the legacy systems. There is no bridge. Modules register hooks
and services through the old APIs. The new patterns exist in isolation.

The bridge layer makes boot.js instantiate the new patterns, register them as services in BOTH
systems, and wraps the old APIs to forward to the new ones. This allows gradual migration —
existing modules keep working while new code uses the new patterns.

## Read These Files First
- core/boot.js — the boot sequence orchestrator (understand the full init flow)
- core/hooks.js — legacy hook system being bridged
- core/services.js — legacy service registry being bridged
- core/lib/Hook/HookManager.js — new hook system
- core/lib/DependencyInjection/Container.js — new container
- core/lib/Plugin/PluginManager.js — needs infrastructure wiring
- core/lib/Entity/EntityTypeManager.js — needs infrastructure wiring (if built)

## Code Style Rules
- WHY comments explaining bridge decisions
- Full JSDoc
- Descriptive deprecation warnings with migration guidance
- Zero npm dependencies

## Technology Stack
- Runtime: Node.js (zero npm dependencies)
- Port: 3001

## Phase 1: Bridge Layer (Features 1-4)

1. Create core/lib/Bridge/ServiceBridge.js that wraps the legacy core/services.js to forward registrations to the new Container. Constructor takes (legacyServices, container). Implement register(name, factory, deps) that registers in BOTH the legacy services object AND the new Container. Implement get(name) that tries new Container first, falls back to legacy. Implement wrapLegacy() that iterates all existing legacy service registrations and registers them in the Container. Log a deprecation notice on legacy-only access.
   ACCEPTANCE: ServiceBridge.register('test', factory) registers in both systems. get('test') resolves from Container. wrapLegacy() migrates existing legacy services.

2. Create core/lib/Bridge/HookBridge.js that wraps legacy core/hooks.js to forward to new HookManager. Constructor takes (legacyHooks, hookManager). Implement register(hookName, handler, options) that registers in BOTH systems. Implement trigger(hookName, context) that invokes via new HookManager (which runs all handlers including legacy-registered ones). Implement wrapLegacy() that migrates existing legacy hook registrations to HookManager. Log deprecation on legacy-only calls.
   ACCEPTANCE: HookBridge.register('test', handler) registers in both systems. trigger('test') invokes via HookManager. wrapLegacy() migrates existing hooks.

3. Create core/lib/Bridge/index.js barrel export. Implement BridgeManager class that orchestrates the full bridge setup. Constructor takes (legacyServices, legacyHooks). Implement setup() that creates Container, HookManager, ServiceBridge, HookBridge, wires infrastructure on all pattern classes (PluginManager, EntityTypeManager if available), registers the new pattern instances as services in the Container ('hooks', 'container', 'entity_type.manager', 'plugin.manager.*'). Returns {container, hookManager, serviceBridge, hookBridge}.
   ACCEPTANCE: BridgeManager.setup() returns all bridge components. New pattern classes have infrastructure set. Pattern instances are available via container.get('hooks') etc.

4. Create core/lib/Bridge/DeprecationLogger.js utility that tracks and reports legacy API usage. Implement log(apiName, alternative) that records the call with stack trace (first call only per API to avoid spam). Implement report() that returns summary of all deprecated API calls with migration guidance. Implement suppress(apiName) to silence specific warnings during migration. Uses console.warn for output.
   ACCEPTANCE: log('hooks.register', 'Use hookManager.on()') warns once. Second call to same API is silent. report() lists all deprecated calls with alternatives.

## Phase 2: Boot Integration (Features 5-7)

5. Modify core/boot.js to import and instantiate BridgeManager early in the boot sequence (after legacy services and hooks are created but before modules are loaded). Call bridgeManager.setup() to create the bridge layer. Store bridge components on the app/context object so modules can access both old and new APIs. The legacy APIs continue to work — they just forward through the bridge. Add WHY comment explaining the bridge strategy.
   ACCEPTANCE: After boot, both legacy services.get('x') and container.get('x') work for the same service. Modules loaded after bridge setup have their hooks registered in both systems. Boot completes without errors.

6. Wire EntityTypeManager into the bridge. After bridge setup, create EntityTypeManager instance, call setInfrastructure(container, hookManager), register it as service 'entity_type.manager'. Fire entity_type_info hook during boot to discover entity types from modules. If any PluginManager instances exist, wire their infrastructure too. This makes the entity system available to all modules through the standard service container.
   ACCEPTANCE: After boot, container.get('entity_type.manager') returns EntityTypeManager. Modules that register entity types via hook_entity_type_info have their types available.

7. Add a boot phase that validates the Container after all modules are loaded. Call container.validateDependencies() (from cycle detection spec) and log any issues. This catches misconfigured services early rather than at runtime. If cycles are found, log error but don't crash — allow the system to start with a warning.
   ACCEPTANCE: Boot with no cycles completes normally. Boot with a cyclic service dependency logs error with the cycle chain but doesn't crash.

## Phase 3: Barrel Export and Verification (Feature 8)

8. End-to-end verification: Boot the CMS with the bridge layer active. Verify: (1) Legacy hooks.register() still works and handlers fire, (2) New hookManager.on() works and handlers fire, (3) Legacy services.get() resolves services, (4) New container.get() resolves the same services, (5) EntityTypeManager is available and discovers types, (6) A module using the legacy API and a module using the new API can coexist, (7) Deprecation warnings appear for legacy API usage.
   ACCEPTANCE: Full CMS boots successfully with bridge. Both old and new APIs work. No errors in console except deprecation warnings. Server responds on port 3001.
