### Implementation Summary

**Feature #1: ServiceBridge**
The bridge between legacy `core/services.js` and new `Container`:
- Constructor takes `(legacyServices, container)` and stores both references
- `register(name, factory, deps)` registers in BOTH systems for backward compatibility
- `get(name)` tries Container first (forward migration), falls back to legacy
- `wrapLegacy()` migrates existing legacy services to Container
- Logs deprecation warning on legacy-only access (once per service to avoid spam)
- Enables smooth migration: legacy code keeps working, new code uses Container

**Feature #2: HookBridge**
The bridge between legacy `core/hooks.js` and new `HookManager`:
- Constructor takes `(legacyHooks, hookManager)` and stores both references
- `register(hookName, handler, options)` registers in BOTH systems
- `trigger(hookName, context)` invokes via HookManager ONLY (single source of truth)
- `wrapLegacy()` migrates existing legacy hooks to HookManager
- HookManager becomes the execution engine for all hooks (legacy + new)
- Both systems stay in sync during transition

**Feature #3: BridgeManager**
Orchestrates the complete bridge setup:
- Constructor takes `(legacyServices, legacyHooks)`
- `setup()` method performs full bridge initialization:
  1. Creates `Container` and `HookManager` instances
  2. Creates `ServiceBridge` and `HookBridge` adapters
  3. Migrates existing legacy registrations via `wrapLegacy()`
  4. Wires infrastructure on `EntityTypeManager` (if available)
  5. Registers wire infrastructure helper for `PluginManager`
  6. Registers pattern instances as services: `hooks`, `container`, `entity_type.manager`
  7. Returns all bridge components for use in boot.js
- Prevents double setup (throws error if called twice)
- Logs migration summary to console

### Architecture Notes

**Why Container-first lookup?**
ServiceBridge.get() tries Container before legacy because:
- Container has better dependency resolution (automatic recursion)
- Container supports tags, decorators, lazy loading
- Encourages migration to new system
- Still falls back to legacy for compatibility

**Why HookManager-only execution?**
HookBridge.trigger() only invokes HookManager (not legacy) because:
- HookManager already contains ALL handlers (legacy + new) via wrapLegacy()
- Invoking both would run handlers twice
- HookManager is the single source of truth
- Better error handling and features (once-handlers, priorities, modules)

**Why wrapLegacy() at boot?**
Legacy services/hooks may be registered before BridgeManager.setup() runs (early boot phase).
wrapLegacy() migrates them so both systems have the same data from the start.

**Infrastructure wiring pattern:**
Pattern classes (PluginManager, EntityTypeManager) need Container and HookManager references.
BridgeManager wires them during setup, then registers as services for dependency injection.

### Design Decisions

1. **Dual registration strategy**: Register in BOTH systems during transition
   - Ensures backward compatibility
   - Allows gradual migration
   - No breaking changes for existing modules

2. **Deprecation warnings**: Log once per API to guide migration without spam
   - First access logs warning with migration path
   - Subsequent accesses are silent
   - Helps developers identify what needs updating

3. **Migration helpers**: wrapLegacy() provides automated migration
   - Reduces manual work
   - Ensures completeness
   - Logs progress for visibility

4. **Service pattern**: Pattern instances are themselves services
   - Available via dependency injection: `container.get('hooks')`
   - Enables clean module architecture
   - Follows Drupal's container pattern
