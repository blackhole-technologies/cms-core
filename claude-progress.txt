### Architecture Notes

**Design Philosophy: CTools (Developer Utilities)**
These three features form the foundation for building advanced CMS features:
- Plugin Type Manager enables plugin discovery and validation
- Typed Data provides form validation and API contracts
- Conditions enable flexible rules for blocks, workflows, access control

**Service Registration Pattern**
All three services follow the dual registration pattern:
```javascript
export function register(services, container) {
  // Legacy services.register()
  services.register('service_name', () => api);

  // New Container.register()
  container.register('service_name', () => api, {
    tags: [...],
    singleton: true,
  });
}
```

**Why These Features Matter**
1. **Plugin Type Manager**: Essential for building extensible systems. Field types, block types, formatter types all need type management.
2. **Typed Data**: Critical for form validation, API validation, and data integrity. Prevents invalid data from entering the system.
3. **Conditions**: Powers conditional visibility, workflow rules, access control, and smart content features.

### Architecture Notes

**Template Token System**
The template system uses simple string replacement with escaping:
- `{label}` → entity label (escaped)
- `{id}` / `{value}` → entity ID (escaped)
- `{color}` → color value (escaped)
- `{icon}` → raw HTML/emoji (not escaped, for flexibility)

**Color Application Strategy**
Two approaches supported:
1. **Direct color property**: Tag data includes `color: "#ef4444"`, applied via CSS variable
2. **Entity field**: `tagColorField: "field_color"` setting fetches color from entity

**Icon Support**
Icons can be:
- Emoji characters (🟨 🐍 💎 🔵)
- Unicode symbols
- SVG markup (inline)
- Image tags (for icon files)

**CSS Class Naming Convention**
- Vocabulary: `.tagify__tag--vocab-{vocabulary_id}`
- Entity type: `.tagify__tag--type-{entity_type}`
- Colored: `.tagify__tag--colored`
- Icon wrapper: `.tagify__tag-icon`

### Architecture Notes

**Profile-Based Configuration**
The Linkit module now supports multiple configuration profiles that can be selected based on context (user role, content type, field, etc.). This enables:
- Different autocomplete behavior for different user roles
- Content-only profiles for basic editors
- Full access profiles for administrators
- Custom profiles for specific use cases

**CSRF Protection**
Implemented proper CSRF protection for state-changing operations:
- Tokens generated per-session via auth service
- Sent via X-CSRF-Token header in AJAX requests
- Meta tag embedding for JavaScript access
- Admin middleware validates tokens automatically

**Drupal Parity**
This implementation mirrors Drupal's Linkit module profile system, providing:
- Multiple named profiles with independent configurations
- Entity type filtering per profile
- Matcher-specific settings
- Role-based access control
- Profile import/export capability (foundation)
